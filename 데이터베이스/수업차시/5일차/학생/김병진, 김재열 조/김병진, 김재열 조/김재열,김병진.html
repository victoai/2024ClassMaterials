주제 : 게임상점

  1. mp_server 테이블:
    • 엔티티: 게임 서버
    • 제약 조건:
      • sv_code: primary key
      • sv_name: not null
  1. mp_vip 테이블:
    • 엔티티: VIP 플레이어 등급
    • 제약 조건:
      • grade_code: primary key
      • grade_name: not null
      • grade_dr: 등급 경험치는 0 이상이어야 합니다. (check)
  1. item_cubetbl 테이블:
    • 엔티티: 아이템 큐브
    • 제약 조건:
      • item_code: primary key
      • item_name: not null, unique
      • up_percent: 아이템의 강화 확률은 0보다 커야 합니다. (check)
      • item_price: not null
  1. mp_wp 테이블:
    • 엔티티: 게임 내 무기 유형
    • 제약 조건:
      • wp_code: primary key
      • wp_cat: '앱솔랩스', '파프니르', '아케인', 또는 null 중 하나여야 합니다. (check)
      • wp_price: 기본 값은 0이며, 0 이상이어야 합니다. (check)
  1. mp_usertbl 테이블:
    • 엔티티: 사용자 데이터
    • 제약 조건:
      • user_id: primary key
      • grade: mp_vip 테이블의 grade_code를 참조하는 외래 키 (foreign key)
      • sv_code: mp_server 테이블의 sv_code를 참조하는 외래 키 (foreign key)
  1. mp_store 테이블:
    • 엔티티: 상점에서의 사용자 주문
    • 제약 조건:
      • order_id: primary key
      • user_id: mp_usertbl 테이블의 user_id를 참조하는 외래 키 (foreign key)
      • item_code: item_cubetbl 테이블의 item_code를 참조하는 외래 키 (foreign key)
      • wp_code: mp_wp 테이블의 wp_code를 참조하는 외래 키 (foreign key)
      • cnt: 주문 수량은 0보다 커야 합니다. (check)

테이블 활용 예제

SUM 대신에 MAX 함수를 사용해야 하는 이유는 각 주문에 대한 항목마다 합계를 구하는 것이 아니라, 주문당 하나의 항목에 대한 최대값을 가져와야 하기 때문입니다.

아이템코드와 무기코드를 따로 받아오기 때문에 case문을 이용하여 각 코드별 null값이 아닌 code의 총액을 구하기 위해 is not null을 이용했습니다.

주문 테이블의 각 주문은 하나의 상품 또는 무기와 연결되어 있습니다. 하지만 모든 주문이 상품 또는 무기와 연결되어 있지는 않기 때문에 left join을 사용했습니다.
